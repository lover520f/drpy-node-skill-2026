# drpy-node Source Workflow

> drpy-node 的总控入口 skill。
> 目标：**先评估，再分流，最后汇总，不自己承担所有细节。**

---

## 本 skill 只做三件事

1. **判断任务类型**
2. **先做基线评估**
3. **决定分流与汇总**

如果任务已经明确属于播放、建源或上传，不要把细节全部留在本 skill 里展开。

---

## 适用场景

当用户出现这些意图时触发：
- 修某个 drpy 源
- 测试某个源是否可用
- 详情为空 / 不能访问详情
- 播放地址不对 / lazy 不通
- 搜索异常 / 分类异常 / 首页无数据
- 上传源到仓库
- 修复 P0 / P1 异常源
- 创建新的 DS 源

---

## 与子 skill 的关系

### 本 skill 是主入口
它负责：
- 接住任务
- 先评估
- 分流
- 汇总

### 子 skill 负责专项执行
- `drpy-node-play-debug` → 播放 / lazy / 直链 / 假播放
- `drpy-node-source-create` → 新建 DS 源
- `drpy-node-repo-upload` → 仓库上传 / 标签 / 替换上传

### 使用原则
- **本 skill 负责判断，不负责把所有专项细节都写满。**
- **专项问题进入子 skill 的思路处理。**

---

## 标准入口动作（硬规则）

遇到修源 / 测源 / 上传相关任务时，**第一动作默认应是评估，而不是直接改文件或直接上传。**

推荐起手：
1. 确认源名或目标站点
2. `list_sources`（修已有源时）
3. `evaluate_spider_source`
4. 必要时 `test_spider_interface`

---

## 任务类型判断

### A. 普通修源
典型表现：
- 首页异常
- 一级异常
- 二级为空
- 搜索异常

→ 本 skill 可先评估，再决定是否继续本地修复。

### B. 播放专项
典型表现：
- 播放不通
- lazy 不对
- 是不是直链
- `play.html` 被当直链
- 假播放

→ 应优先切换到 **`drpy-node-play-debug`** 的思路。

### C. 上传专项
典型表现：
- 上传仓库
- 替换上传
- 改标签
- 上传前检查

→ 应优先切换到 **`drpy-node-repo-upload`** 的思路。

### D. 新建源
典型表现：
- 新建源
- 写个 drpy 源
- 分析站点做 DS 源

→ 应优先切换到 **`drpy-node-source-create`** 的思路。

---

## 问题分层判断（用于决定后续走向）

### 首页层
- 没分类
- 没推荐
- 返回空数据

### 一级层
- 没有 `vod_id`
- 没有 `vod_name`
- 没有 `vod_pic`
- 列表为空

### 二级层
- `detail` 返回空
- 无 `vod_name`
- 无 `vod_play_from`
- 无 `vod_play_url`

### 搜索层
- 搜不到结果
- 返回结构错误

### 播放层
- `lazy` 不返回结果
- 返回中间标识符
- 把 `play.html` 当直链
- parse 判断错误
- 不是真正的 m3u8/mp4/http 可播地址

---

## 分流规则（精简版）

### 进入 `drpy-node-play-debug`
满足任一：
- 用户明确只关心播放 / lazy
- 详情正常但播放不正常
- 出现 `play.html`
- 出现 `JD-xxxx` 这类中间标识
- 需要判断直链 / 解析页 / 假播放

### 进入 `drpy-node-repo-upload`
满足任一：
- 用户目标是上传
- 需要替换上传
- 需要改标签

### 进入 `drpy-node-source-create`
满足任一：
- 用户要从零新建源
- 需要模板判断
- 当前不是修已有源，而是分析新站点

---

## 结果汇总时必须说明的事

无论是否分流，最后都要清楚说：
- 数据层是否修好
- 播放层是否真正修好
- 当前是否满足上传条件
- 如果未上传，卡点是什么

### 特别注意
以下情况必须主动说明，不要模糊带过：
- 评估器通过，但播放不是真直链
- 详情恢复了，但播放还没彻底好
- 标签还没确认，暂不适合上传

---

## 上传决策规则

### 可以考虑上传
满足：
- 语法通过
- 结构通过
- 至少首页 / 一级 / 二级正常
- 用户明确要求上传

### 不应直接上传
满足任一：
- 详情为空
- 一级字段不规范
- 播放明显是假通过，但用户要求“修好后再传”
- 只是猜测修复，未复测

---

```markdown
## 当前结论

- 首页：...
- 一级：...
- 二级：...
- 搜索：...
- 播放：...

## 分流判断
- 是否需要进入播放专项：是 / 否
- 是否满足上传条件：是 / 否
- 下一步建议：...
```

---

## 明确禁止事项

- 不要一上来直接改源码
- 不要一上来直接上传仓库
- 不要把主 skill 写成所有专项细节的大合集
- 不要在播放问题上跳过 `drpy-node-play-debug`
- 不要在上传问题上跳过 `drpy-node-repo-upload`

---

## 最终目标

让 drpy-node 相关任务形成更稳定的协作闭环：

**入口判断 → 评估 → 分流 → 专项执行 → 汇总 → 上传决策**
