# drpy-node Source Create

> drpy-node 的新建源专项 skill。
> 目标：**从站点分析出发，按阶段产出“最小可用新源”，而不是一开始就写成大而全成品。**

---

## 本 skill 只负责什么

- 新建 DS 源
- 判断能否继承模板
- 分析目标站点结构
- 按阶段完成分类 / 一级 / 搜索 / 二级 / lazy
- 产出一个可测试的新源

### 不负责什么
- 不负责复杂播放专项深挖
- 不负责仓库上传决策
- 不负责总控调度

---

## 适用场景

当用户说这些时启用：
- 写个新源
- 新建 drpy 源
- 分析这个网站做 DS 源
- 做个可用的规则
- 从零开始写 spider/js/*.js

---

## 核心原则

1. **先分析站点，再决定模板还是手写。**
2. **先做分类/一级，再做搜索，再做二级，再做 lazy。**
3. **不要一上来就写完整 lazy，先保证数据链路通。**
4. **现代前端站点不要乱套旧模板。**
5. **新源完成后必须做语法、结构、接口测试。**
6. **优先产出“最小可用新源”，再继续增强。**
7. **初版已经可用时，要知道适时停手，而不是继续过度开发。**

---

## 什么叫“最小可用新源”

一个新源最少应该做到：
- 有分类
- 一级可出列表
- 二级可出详情
- 搜索至少有一种可行方案
- lazy 暂时可以简单返回合理结果

### 不算完成的情况
- 只有分类，没有列表
- 只有一级，没有二级
- 二级没有 `vod_play_from / vod_play_url`
- lazy 完全未定义，但却宣称“新源已完成”

---

## 站点类型先分两类

### A. DOM 驱动站点
特征：
- HTML 里就能看到影片列表
- 页面源码包含卡片、标题、图片、链接
- 不依赖前端 JS 才能看到主要内容

### B. API 驱动站点 / 现代前端站点
特征：
- 页面 HTML 几乎是空壳
- 真实数据来自 `/api/...` JSON 接口
- React / Vue / Tailwind 明显
- 内容靠前端渲染

**判断原则：**
- DOM 驱动 → 可优先写选择器 / 尝试模板
- API 驱动 → 优先手写请求 JSON，不要执着于 DOM 规则

---

## 模板判断规则（第二轮强化版）

### 可以尝试模板
如果站点明显是传统 CMS：
- `首图`
- `mx`
- `mxpro`

### 明确不建议继承模板
满足任一就要高度警惕：
- React / Vue SPA
- Tailwind 类名大量出现
- 数据主要来自 JSON API
- HTML 中没有有效影片卡片内容
- 依赖前端脚本渲染列表

### 模板误判反例
以下情况最容易误套模板：
1. 页面表面有列表容器，但内容其实是前端渲染的占位壳
2. 首页 DOM 和分类 DOM 长得像 CMS，但真实数据来自接口
3. 搜索结果并不走列表页结构，而是独立 suggestion API

### 一个实用判断原则
如果你分析页面后发现：
> 真正有价值的数据在 API，不在 HTML DOM

那就应该优先考虑：
- 手写规则
- 直接请求 API
- 放弃旧模板继承

## 标准创建流程

### Step 1：模板/站点类型判断
优先用：
- `guess_spider_template`
- `fetch_spider_url`
- `analyze_website_structure`

### Step 2：先做分类
优先方案：
- `class_name + class_url`
- 或 `class_parse`

### Step 3：先做一级列表
一级目标：
- `vod_id`
- `vod_name`
- `vod_pic`
- `vod_remarks`

建议：
- 列表先用兼容字段组装
- 再 `setResult(d)`

### Step 4：再做搜索
判断：
- 是否能复用一级结构
- 是否有独立搜索 API
- 是否适合用 `搜索: '*'`

### Step 5：再做二级详情
二级最少应包含：
- `vod_name`
- `vod_pic`
- `vod_play_from`
- `vod_play_url`

### Step 6：最后做 lazy
优先级：
1. 直链
2. 站外解析页
3. iframe
4. 中间脚本变量
5. 加密逻辑

---

## 分阶段开发策略

### 第一阶段：打通最小数据链路
先确保：
- 分类正常
- 一级正常
- 二级正常

### 第二阶段：补搜索
- 如果和一级结构相同，优先复用
- 如果搜索接口单独存在，单独接入

### 第三阶段：补 lazy
- 不要让 lazy 阻塞整个建源进度
- 初版 lazy 可以先返回合理站外解析页
- 后续再深挖直链

### 第四阶段：增强与优化
- 过滤器
- 多线路
- 更精准搜索
- 更优播放链路

---

## 什么时候该停下来

如果已经满足：
- 首页正常
- 一级正常
- 二级正常
- 搜索有基本可用方案
- lazy 至少不会明显误判

那么这时可以先定义为：

> **初版可用新源**

不要因为“还可以继续优化”就立刻把：
- 高级 lazy
- 全量过滤器
- 所有线路修到极致

全部一次性写完。

### 适时停手的好处
- 先拿到可测成果
- 降低一次性写崩的概率
- 方便后续按问题继续细化

## 新源完成后必须做的验证

1. `check_syntax`
2. `validate_spider`
3. `test_spider_interface`
4. `evaluate_spider_source`

### 推荐最低通过标准
- 首页正常
- 一级正常
- 二级正常

如果用户特别关心播放，则继续完善 lazy 后再宣称“新源完成”。

---

## 推荐工具顺序

1. `guess_spider_template`
2. `fetch_spider_url`
3. `analyze_website_structure`
4. `get_spider_template`
5. `debug_spider_rule`
6. `write_file`
7. `check_syntax`
8. `validate_spider`
9. `evaluate_spider_source`

## 明确禁止事项

- 不要站都没分析就盲套模板
- 不要先写 lazy 再补列表
- 不要 HTML 几乎没数据还硬写 DOM 规则
- 不要未校验就说“新源已完成”
- 不要把“只有首页和一级”误说成“源已经做完”
