# drpy-node Play Debug

> drpy-node 的播放专项 skill。
> 目标：**把 lazy 修到三种明确状态之一：直链 / 合理解析页 / 明确未修好。**

---

## 本 skill 只负责什么

- lazy 修复
- parse:0 / parse:1 判断
- 直链识别
- `play.html` 误判排查
- 站外线路筛选
- 假播放识别

### 不负责什么
- 不负责新建源流程
- 不负责仓库上传决策
- 不负责总控调度

---

## 适用场景

当用户说这些时启用：
- 播放不通
- lazy 有问题
- 返回的不是直链
- parse:0 / parse:1 怎么判断
- `play.html` 被当直链
- 站外线路怎么处理
- iframe 怎么提取
- m3u8 / mp4 怎么抠
- 评估器通过但我怀疑是假播放

---

## 核心原则

1. **不要把非空返回误判为播放正常。**
2. **不要把 `play.html` 页面当成直链。**
3. **直链优先于解析页，解析页优先于中间标识。**
4. **原站主线路解不动时，优先看站外聚合线路。**
5. **必须区分“评估器通过”与“真实可播”。**
6. **如果继续深挖的收益明显低于站外替代线路，应及时止损。**

---

## 三种结果判定（最终只允许落到这三类）

### A. 真实直链
满足任一：
- `https://...m3u8`
- `https://...mp4`
- `https://...flv`
- `https://...m4s`
- 其他明确可直接播放的媒体链接

返回应为：
```js
{ parse: 0, url: '真实媒体地址' }
```

### B. 合理解析页
满足任一：
- 腾讯 / 爱奇艺 / 优酷 / 芒果 / B站等页面地址
- 资源站 `play.html`，但尚未抠出真实媒体链接
- iframe 中指向的仍是页面，不是媒体文件

返回应为：
```js
{ parse: 1, url: '页面地址' }
```

### C. 明确未修好 / 假播放
以下都不能说“播放修好”：
- `JD-xxxx`、`ABCD-xxxx` 等中间标识
- `play.html` 却返回 `parse: 0`
- 空地址
- 非视频资源链接
- lazy 返回结构不完整

---

## 标准排查流程

### Step 1：确认二级播放字段
先测 `detail`，确认：
- `vod_play_from`
- `vod_play_url`
是否存在。

### Step 2：抽样策略
至少抽两类：
1. 第一条主线路
2. 一条站外线路

如果站外线路很多，优先抽：
- 腾讯 / 爱奇艺 / 优酷 / 芒果 / B站
- 再抽量子 / 非凡 / 暴风 / 极速 / 如意 / tt

### Step 3：单测 `play`
用 `test_spider_interface(interface='play')` 测 lazy 返回。

### Step 4：按决策树判断
不要凭感觉判断，按下面决策树走。

---

## 播放决策树（第二轮强化版）

```text
输入 play_url
   ↓
是否已经是 m3u8/mp4/flv/m4s ?
   ├─ 是 → parse:0
   └─ 否
        ↓
是否是腾讯/爱奇艺/优酷/芒果/B站等主流平台地址 ?
   ├─ 是 → parse:1
   └─ 否
        ↓
是否是 /play/*.html ?
   ├─ 是 → 进入页面解析流程
   └─ 否
        ↓
是否是中间标识/token ?
   ├─ 是 → 明确标记为未修好
   └─ 否 → 视情况判为 parse:1 或继续分析
```

---

## `play.html` 页面解析流程

遇到：
```text
https://xxx.com/play/123-1-1.html
```

按这个顺序来：

### 第一层：先看 iframe
优先工具：
- `extract_iframe_src`

#### 判断规则
- iframe 指向 m3u8/mp4 → `parse: 0`
- iframe 指向腾讯/优酷等平台页 → `parse: 1`
- iframe 指向新的解析页 → 继续分析

### 第二层：再看 `player_aaaa`
如果页面里存在：
- `player_aaaa`
- `player_data`
- `var player = ...`

优先从中提取：
- 真实视频地址
- 中间地址
- iframe 地址

#### 判断规则
- 提取到媒体文件 → `parse: 0`
- 提取到页面地址 → `parse: 1`
- 提取到 token / 标识符 → 仍未修好

### 第三层：再找脚本里硬编码的 m3u8/mp4
如果页面脚本中直接包含：
- `m3u8`
- `mp4`
- `source`
- `url`

优先直接提取。

### 第四层：兜底
如果页面分析失败：
```js
{ parse: 1, url: playHtml }
```

不要：
```js
{ parse: 0, url: playHtml }
```

## 站外线路优先级

### 第一优先级：主流平台线路
例如：
- 腾讯
- 爱奇艺
- 优酷
- 芒果
- B站

原因：
- 逻辑最清晰
- `parse: 1` 即可
- 最不容易误判

### 第二优先级：资源站线路
例如：
- 量子
- 非凡
- 暴风
- 极速
- 如意
- tt

处理原则：
- 能抠出 m3u8 就 `parse: 0`
- 只能拿到页面就 `parse: 1`
- 绝不把页面误写成 `parse: 0`

### 第三优先级：原站私有线路
例如：
- `JD-xxxx`
- 站内资源 token

如果没完全解开：
- 要明确告诉用户“还没修到真实可播”
- 不要伪装成已完成

---

## 止损规则（新增）

如果出现以下情况，应优先建议“使用站外线路先落地”，而不是继续死磕主线路：
- 原站主线路明显依赖 wasm / protobuf / 私有签名
- 站外聚合线路已经存在且可返回合理解析页
- 用户目标是“先能播”，不是“必须解开主站私有协议”

也就是说：

> **能用站外线路先完成可播，就不要无休止深挖主站私有解密。**

## 常见错误与修正

### 错误 1：把 `play.html` 当直链
错误：
```js
{ parse: 0, url: 'https://xx.com/play/1-1-1.html' }
```

修正：
- 先解析页面
- 不行就：
```js
{ parse: 1, url: 'https://xx.com/play/1-1-1.html' }
```

### 错误 2：把 token / 标识符当播放成功
错误：
```js
{ parse: 0, url: 'JD-xxxx' }
```

修正：
- 如果没解出 http 媒体地址，就不能说修好了

### 错误 3：只看评估器分数
错误认知：
- `play` 成功 → 播放修好

修正：
- 必须看返回 URL 类型
- 必须解释这是直链、解析页还是中间标识

## 明确禁止事项

- 不要把 `play.html` 当成直链
- 不要把中间标识说成可播地址
- 不要因为评估器给通过就说播放彻底修好了
- 不要一上来就先啃 wasm / protobuf，而忽略站外聚合线路
